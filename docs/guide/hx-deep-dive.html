Phases

工作流阶段

CLI 以阶段组织能力，而非单个命令。

Phase 01-03

准备输入

/hx-doc、/hx-plan 负责准备输入；/hx-ctx 是可选的预检诊断工具。

Phase 04-06

执行与校验

/hx-run、/hx-qa、/hx-review、/hx-fix 负责执行与复核。

Phase 07-08

沉淀与交付

/hx-clean 与 /hx-mr 负责沉淀与交付。

Runtime

运行时路径

框架将资源划分为系统层、用户层和项目层，Claude Code 在运行时按优先级路由。

层级 典型路径 角色
FRAMEWORK_ROOT src/agents/commands/（命令实体）
src/profiles/base/（内置基础 Profile）
src/pipelines/（内置流水线）		 框架内置资源（只读），通过 git pull 升级。命令不直接在此运行，由转发器路由。
USER_HX_DIR ~/.hx/commands/（用户命令覆盖）
~/.hx/profiles/（用户 Profile）
~/.hx/pipelines/（用户流水线）
~/.hx/config.yaml		 跨项目共享的用户级覆盖。由 hx setup 创建目录结构，用户自行写入覆盖文件。
PROJECT_ROOT .hx/commands/（项目命令覆盖）
.hx/profiles/（项目 Profile）
.hx/pipelines/（项目流水线）
.hx/config.yaml		 项目专属覆盖，优先级最高。由 /hx-init 初始化 .hx/config.yaml。
i
配置优先级：内置默认值 → ~/.hx/config.yaml → .hx/config.yaml（字段级深合并，数组整体覆盖）。
i
转发器机制：~/.claude/commands/ 中的 hx-*.md 是转发器，不含命令逻辑。Claude Code 运行 /hx-* 时，转发器按优先级查找实体文件（项目层 → 用户层 → 系统层），读取并执行。
Profiles

Profile 解析

Profile 决定架构层级、门控命令、黄金规则与模板资源。当前框架内置只保留 base，其余统一走用户级或项目级自定义 Profile。

搜索顺序

Profile 搜索根目录

<project>/.hx/profiles/
~/.hx/profiles/
框架内置 profiles/base/

同名 Profile 按该顺序覆盖。

推断顺序

hx run 如何确定 profile

--profile 参数
→ .hx/config.yaml 的 defaultProfile
→ ~/.hx/config.yaml 的 defaultProfile

Claude 命令运行时按此顺序确定使用哪个 Profile。

!
参数范围：只有读取 Profile 的命令需要 --profile；hx version 与 hx setup 不依赖该参数。
Context

上下文装配

/hx-run 通过固定清单组装上下文与 Hook。

主上下文

/hx-run 默认读取的文件

src/profiles/base/golden-rules.md
当前 Profile 的 golden-rules.md
当前 Profile 的 profile.yaml
paths.planDoc 对应文件
paths.requirementDoc 对应文件

路径由 .hx/config.yaml 的 paths.* 模板与 Profile 解析结果决定。

Hook 注入

两种定义方式，统一合并

hooks:
  run:
    pre:
      - .hx/hooks/extra.md

加载顺序为全局约定 → 项目约定 → 配置式数组；相同绝对路径会去重。

Quality

质量控制

质量控制由多个独立命令组成，可按场景组合执行。

静态校验

/hx-ctx

/hx-ctx 面向当前需求做轻量预检，主要用于排查文档、计划、进度文件或 Profile 资源缺失。

质量校验

/hx-qa

按固定顺序执行 lint → build → type → test，只跑 Profile 里实际定义过的步骤，遇到失败立即停止。

复核修复

/hx-review / /hx-fix

review 输出检查清单；fix 根据日志或测试输出生成修复 Prompt。

!
占位符参数：若门控命令包含 {scheme} 等占位符，必须通过 CLI 参数显式传入。
Progress

计划与进度

多个命令依赖 paths.progressFile 模板对应的 progress JSON 进行状态推断。

计划生成

/hx-plan 输出两份文件

输出路径由 .hx/config.yaml 的 paths.planDoc 和 paths.progressFile 模板决定。

状态流转

任务状态至少涉及这些值

pending
in-progress
done

任务状态默认由 /hx-run 自动更新；/hx-ctx 负责检查任务定义和进度文件完整性。

Pipeline

流水线编排

/hx-go 由流水线 YAML 驱动，支持三层覆盖和断点恢复。

查找顺序

Pipeline YAML 三层路径

<project>/.hx/pipelines/default.yaml
~/.hx/pipelines/default.yaml
框架内置 src/pipelines/default.yaml

找到第一个存在的文件即使用，可按层级覆盖系统默认流水线。

步骤定义

YAML 步骤字段

steps:
  - id: doc
    phase: Phase 01
    name: 需求文档
    command: hx-doc
    checkpoint:
      message: 确认后继续
  - id: ctx
    command: hx-ctx
    on_fail: stop

checkpoint 暂停等待用户确认；on_fail: stop 失败时终止流水线。

✓
断点恢复：使用 /hx-go user-login --from run --profile base 从指定步骤继续，跳过已完成的 doc/plan 阶段。
